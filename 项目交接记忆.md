## 1. 项目目标

这是一个 Windows WinForms 工具，核心目标是：

- 截取 VMware/游戏窗口画面。
- 使用 Depth Anything ONNX 模型生成深度图。
- 根据深度图判断前方是否可通行。
- 使用开发板发送 `W/A/S/D/跳跃` 等键盘控制。
- 使用 UNIFY + msdk 双头鼠标控制视角旋转。
- 做简单的视觉寻路测试：前方能走就走，不能走就转向，卡住就跳或重新 360 度扫描。
- 同时保留 YOLO 攻击推理测试入口。

项目当前不是完整导航系统，更像是“基于深度图的局部自动前进/避障实验”。

## 5. 当前 UI 默认值

大致默认值：

```text
窗口句柄：自动查找，也曾默认 329860
障碍阈值：65
前方暗区阈值：78
寻路直行阈值：45
寻路转向阈值：55
COM 口：COM5
VMware 端口：5901
```

阈值含义：

- 障碍阈值：深度判断是否通过的主阈值。
- 前方暗区阈值：判断天空/虚空/深度不可靠画面的阈值。
- 寻路直行阈值：寻路中认为可以继续前进的障碍比例阈值。
- 寻路转向阈值：超过后寻路开始尝试转向。

## 6. 深度图判断逻辑

深度推理流程：

1. 截图窗口。
2. 中心裁切正方形。
3. 缩放到模型输入。
4. ONNX Runtime 推理。
5. 得到 float 深度图。
6. 用 Inferno 色图显示。
7. 用原始 float 深度图计算障碍比例。

障碍比例计算：

- 取深度图下半部分作为地面候选区域。
- 计算该区域深度 30% 分位数。
- 全图中深度值大于等于该分位数的像素视为障碍。
- 统计比例得到 `obstaclePercent`。

通行显示逻辑：

```text
障碍比例 > 障碍阈值         => 无法通过
前方暗区异常                => 可能无法通过
障碍比例 >= 阈值 - 10       => 谨慎通过
其他                        => 可以通过
```

## 7. 寻路逻辑现状

入口：

```csharp
BtnTestPath_Click
PathTestLoop
```

当前寻路已经不是简单转向，而是：

1. 开始后等待 5 秒，方便用户把鼠标焦点移入游戏窗口。
2. 读取当前深度图。
3. 根据寻路直行/转向阈值判断是否前进或转向。
4. 前方可走时按住 `W`。
5. 障碍较高或暗区异常时松开 `W`，右转寻找出口。
6. 连续受阻会 `S` 后退。
7. 多次转向失败后做 360 度扫描。
8. 360 扫描时分 24 个方向采样，选择障碍分数最低的方向。
9. 卡住时触发跳跃键 `j`。

360 扫描参数：

```csharp
scanSteps = 24
fullTurnUnits = 2064
stepUnits = fullTurnUnits / scanSteps
```

方向评分：

```text
score = 障碍比例 + 暗区异常惩罚 + 前方暗区超阈值惩罚
```

注意：

- `2064` 是当前经验值，用于大约旋转一圈。
- 如果实际游戏里一圈不足或过头，需要调 `fullTurnUnits`。

## 13. 当前注意事项

1. 源码里仍能看到部分中文乱码字符串。

   这多半是历史编码问题，不一定影响编译，但后续最好逐步清理为 UTF-8 中文。

2. 寻路参数还在实验阶段。

   重点可调：

   ```text
   寻路直行阈值
   寻路转向阈值
   fullTurnUnits
   scanSteps
   SmoothTurnMouse 步数和延迟
   卡住判断帧数
   ```

3. 深度判断不是地图寻路。

   它只看当前画面，适合局部避障，不保证能走到目标点。

4. YOLO 攻击和深度寻路是两套逻辑。

   YOLO 依赖 `UNIFY.dll`、`whatlan.onnx` 和 VMware 连接。

5. msdk/UNIFY 初始化失败时，优先检查：

   ```text
   data\UNIFY.dll
   data\msdk.dll
   COM 口
   VMware 端口 5901
   UNIFY COM 注册
   程序 x64/x86 与 DLL 位数是否匹配
   ```

## 14. 接手建议

优先级建议：

1. 先确认项目能构建。
2. 确认深度模型是 CUDA/TensorRT 而不是 CPU。
3. 确认窗口句柄自动获取有效。
4. 确认截图和深度图显示正常。
5. 确认开发板 `W/S/A/D/J` 正常。
6. 在游戏里把跳跃绑定到 `J`。
7. 再测试寻路。
8. 最后再接 YOLO 攻击逻辑。

如果要继续优化寻路，建议下一步做：

- 加一个“当前方向评分曲线”日志。
- 记录 360 扫描的每个方向障碍比例。
- 把 `fullTurnUnits` 做成界面可调。
- 增加“是否真的移动”的判断，比如比较连续截图中心画面的变化。
- 增加左前/右前区域深度分析，让转向更智能。

## 15. 重要结论

当前最关键的工程决策：

```text
跳跃不再发空格，改发 J。
```

原因是开发板空格 HID 表现异常，而普通字母键稳定。

接手者如果发现跳跃不工作，第一件事不是查 C#，而是检查游戏内是否把跳跃绑定到了 `J`。
